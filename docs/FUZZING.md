# Fuzzing

## Strategy

QRD uses fuzz testing to validate parser safety, reader resilience, and format robustness.

### Primary Targets

- file header parsing
- footer parsing and checksum validation
- row group and column chunk parsing
- schema deserialization
- compression/decompression boundaries
- encryption and ECC metadata handling

### Goals

- detect malformed input handling bugs
- validate failure modes for truncated or corrupted files
- ensure memory-safe parsing across bindings
- document reproducible crash cases for regression fixes

## Corpus

A robust corpus should include:

- valid QRD files generated by the engine
- intentionally malformed headers and footers
- truncated or malformed row groups
- invalid metadata values and unknown extensions
- files with unsupported version combinations

## Test Harness

- use a Rust-based fuzz harness when possible
- integrate fuzz targets with the existing test framework
- preserve minimized inputs for regression coverage

## CI Integration

- include fuzz regression or corpus validation in CI when feasible
- run a small, deterministic subset of fuzz cases for each relevant change
- document any performance or stability findings in the PR

## Memory Safety

- the Rust core is the reference implementation.
- fuzzing helps detect logic bugs, invalid assumptions, and unsafe behavior.
- any discovered unsafe code path should be reviewed and documented.

## Reporting

- log the fuzz target and corpus file for each found issue
- add a regression test when a bug is fixed
- update documentation if the issue exposes a spec ambiguity
